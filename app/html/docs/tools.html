Flint Build Tools

Overview

The Flint build tools take care of quite a few tasks making it easy to build your application without the need to restart the server or recompile files while you're working.

The following diagram illustrates how the basic components work to create an application:

The core classes are highlighted in blue. There are three primary languages used to construct Flint applications. You should familiarize yourself with these languages before getting started:

  1. Coffeescript is a language that compiles into Javascript with all sorts of syntax sugar.
  2. Handlebars is a "logicless" template language that keeps your templates clean.
  3. Stylus (not in the diagram) is a CSS compiler like LESS, SASS etc.

Using The Command Line Tool

The command line tool will print a manual page by simply entering the flint command in your terminal. Here are the arguments explained in detail:

--new (-n)

The new command creates a new Flint application. To start a new app type:

flint -n todos

Before you use the rest of the command line tools cd into your new application: 

cd todos

--server (-s)

The server command spins up an express server instance which you can view in your browser at http://localhost:3000 

flint -s

NOTE: You may have to install the request module globally even though it is part of express

--watch (-w)

Watches all your application files and recompiles when changes are detected. This avoids having to recompile to see the result of your changes.

flint -w

Note: To watch changes to your Responder classes, you will also need to pass --server (-s) to the flint command.

--quiet (-q)

This is a watch modifier which tells flint to only print compile errors to the console. By default, console messages are sent each time a change is detected to a file.

flint -wq

--compile (-c)

Compiles client side files to their respective outputs once and completes without watching. Useful for quick changes to copy that may not require testing/viewing.

flint -c

--package (-p)

Packages and compresses the "nice" compiled output into a single minified build target, "production.js" by default.

flint -p

--file (-f) [file]

This argument allows you to specify a configuration file relative to the current working directory to be used in place of the flint.js file. This is very handy for sharing application code and repositories between two desired outputs. (e.g. desktop vs. mobile)

flint -wsf flint-mobile-verion.js

--build (-b)

This command takes compiles the core flint libraries into their targets. The only time this is used is when you require an existing application to use new flint core functionality.

flint -b
